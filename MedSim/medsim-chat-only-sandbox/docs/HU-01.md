## Nombre
HU-01 - Seleccionar paciente e iniciar consulta

## Objetivo
Como estudiante o profesional que practica entrevistas clinicas, quiero elegir un paciente simulado e iniciar una consulta para comenzar una sesion de anamnesis con contexto cargado.

## Contexto
La seleccion de paciente define el caso clinico, el prompt y la continuidad de la sesion. Sin esto no existe contexto para conversar ni para evaluar el cierre.

## Alcance
Incluye:
- Listado de pacientes disponibles.
- Seleccion de paciente y carga de ficha clinica visible.
- Inicio de un `encounter` (consulta) asociado a una sesion (`X-Session-Id`) y a un modo de entrevista.

No incluye:
- Autenticacion/roles (es un sandbox local).
- Persistencia en DB (estado en memoria y pacientes en JSON).

## Reglas de negocio
- Un `encounter_id` pertenece a una sola sesion (header `X-Session-Id`).
- Si se selecciona un paciente, se inicia un nuevo `encounter` para ese paciente y modo.
- Si no hay pacientes, no se puede iniciar una consulta.

## Criterios de aceptacion
- Dado que hay pacientes cargados Cuando abro la UI Entonces puedo seleccionar un paciente.
- Dado un paciente seleccionado Cuando se inicia el encounter Entonces la ficha clinica basica queda visible.
- Dado un modo elegido (Libre/SEGUE) Cuando inicio la consulta Entonces el `encounter` queda asociado al modo.

## Casos borde y errores
- No hay pacientes en disco: mostrar mensaje y bloquear inicio.
- Paciente inexistente o ID invalido: devolver 404/400 con mensaje claro.
- Falta `X-Session-Id`: el backend genera un session id derivado (no recomendado pero soportado).

## Datos y validaciones
Entidades/atributos:
- `PatientProfile` (JSON): `id`, `name`, `age`, `region`, `chief_complaint`, `known_medical_history`, etc.
- `Encounter`: `encounter_id`, `patient_id`, `mode`, `started_at`, `finished_at`.

Validaciones:
- `patient_id` debe existir y corresponder a un JSON valido.
- `mode` debe estar en `{free, segue}`; si no, se normaliza a `free`.

## Seguridad y roles
- Sin autenticacion (sandbox). El aislamiento es por `X-Session-Id`.

## Dependencias e integraciones
- Lectura de pacientes desde `patients/*.json`.
- Prompt del paciente generado desde `services/prompt_service.py`.

## Flujo y UX
- La UI lista pacientes (`GET /api/patients`) y al seleccionar inicia encounter (`POST /api/encounters/start`).
- La ficha clinica se muestra en el panel lateral derecho.

## Observabilidad
- Log de inicio de encounter (recomendado): `session_id`, `encounter_id`, `patient_id`, `mode`.

## Tests sugeridos
Unitarios:
- Normalizacion de modo.
- Fallo si `patient_id` no existe.

Integracion:
- Listar pacientes -> iniciar encounter -> leer encounter.

## Datos de prueba
- `patients/carlos_35_cefalea_mareos.json`
- `patients/jorge_62_dolor_pecho.json`
- `patients/valentina_26_dolor_fosa_iliaca.json`

## Notas
- El historial de chat del encounter vive en memoria (se pierde al reiniciar).
