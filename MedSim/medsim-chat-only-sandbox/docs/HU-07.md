## Nombre
HU-07 - Administrar pacientes base (JSON)

## Objetivo
Como mantenedor del simulador, quiero guardar perfiles de pacientes en archivos JSON para ampliar o ajustar los casos clinicos disponibles sin tocar el flujo principal de la app.

## Contexto
Los pacientes son el contenido del producto. Editarlos como JSON permite iterar rapido sin cambiar codigo.

## Alcance
Incluye:
- Listar pacientes desde disco.
- Leer detalle de un paciente.
- Guardar/actualizar perfil valido en `patients/`.

No incluye:
- Editor UI completo para authoring (se puede agregar, pero no es parte del MVP).
- Versionado/validacion avanzada de casos (mas alla de Pydantic).

## Reglas de negocio
- `patient_id` debe cumplir regex: letras/numeros/`_`/`-`, largo 2-64.
- Guardado es idempotente por `id` (se sobreescribe el JSON con el mismo id).
- Pacientes invalidos se omiten al listar (se loguea warning).

## Criterios de aceptacion
- Dado un JSON valido en `patients/` Cuando listo pacientes Entonces aparece en la UI.
- Dado un JSON invalido Cuando listo pacientes Entonces no rompe el sistema y se omite ese archivo.
- Dado un perfil valido Cuando hago upsert Entonces se guarda y queda disponible.

## Casos borde y errores
- ID invalido: 400 con mensaje claro.
- Directorio inexistente: listar devuelve vacio sin error.
- JSON corrupto: se omite con warning.

## Datos y validaciones
Entidad:
- `PatientProfile` (Pydantic) en `domain/models.py`.

Validaciones clave:
- `id` regex + required fields (nombre, edad, textos clinicos).

## Seguridad y roles
- Sin auth. Recomendado correr en local.

## Dependencias e integraciones
Endpoints:
- `GET /api/patients`
- `GET /api/patients/{patient_id}`
- `POST /api/patients` (JSON body)

## Flujo y UX
- Selector de paciente se alimenta de `GET /api/patients`.
- La ficha clinica muestra datos visibles y secciones "spoiler" (si aplica).

## Observabilidad
- Warnings al cargar pacientes invalidos (archivo + error).

## Tests sugeridos
Unitarios:
- Validacion de `patient_id`.
- Guardado y relectura.

Integracion:
- Upsert -> listar -> leer -> usar en encounter.

## Datos de prueba
- Crear un paciente nuevo con `id` `ana_29_dolor_abdominal`.

## Notas
- Mantener consistencia entre `speaking_style`, `region` y el prompt del paciente.
