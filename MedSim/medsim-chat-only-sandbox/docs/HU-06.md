## Nombre
HU-06 - Verificar salud del sistema

## Objetivo
Como usuario, quiero ver si LLM/STT/TTS estan disponibles para detectar rapido problemas de configuracion antes de iniciar una consulta.

## Contexto
En un sandbox local es comun tener puertos ocupados, keys invalidas o endpoints caidos. Un health check reduce friccion.

## Alcance
Incluye:
- Salud del LLM (models disponibles).
- Diagnostico de STT y TTS via "Test" (configuracion + conectividad basica).
- Indicador visual en UI y logs de diagnostico.

No incluye:
- Observabilidad centralizada (Prometheus/Grafana).
- Alertas automáticas.

## Reglas de negocio
- Los endpoints de estado/config no deben filtrar secretos (solo indican si hay key configurada).
- Un "Test OK" implica conectividad basica; no garantiza calidad del modelo/voz.

## Criterios de aceptacion
- Dado un backend LLM configurado Cuando consulto salud Entonces veo si hay modelos disponibles.
- Dado STT configurado Cuando ejecuto `Test STT` Entonces obtengo un log `Test STT OK` o un error entendible.
- Dado TTS configurado Cuando ejecuto `Test TTS` Entonces se reproduce audio o aparece un error entendible.

## Casos borde y errores
- Timeouts hacia proveedores externos.
- Respuestas no-JSON en endpoints (se captura como `raw`).

## Datos y validaciones
Endpoints:
- `GET /api/llm_health`
- `POST /api/transcribe` (test STT, enviando un WAV simple)
- `POST /api/tts` (test TTS, pidiendo audio de una frase)
- `GET /api/config_state` (estado de configuracion sin filtrar secretos)

## Seguridad y roles
- Sin auth.

## Dependencias e integraciones
- Proveedores externos por HTTP.

## Flujo y UX
- Botones `Test STT` y `Test TTS`.
- Dots de estado (verde/rojo) por servicio.
- Panel `Audio Logs` para diagnostico rapido.

## Observabilidad
- Log de errores con status code y payload crudo cuando falla un proveedor.

## Tests sugeridos
Integracion:
- Mock endpoints externos para simular healthy/unhealthy.

## Datos de prueba
- STT Groq (OpenAI-compatible): base_url + key + model.
- TTS Cartesia: base_url + key + voice_id + model_id.

## Notas
- `schema_version` en `/api/config_state` ayuda a detectar si el frontend esta pegandole a un backend viejo.
